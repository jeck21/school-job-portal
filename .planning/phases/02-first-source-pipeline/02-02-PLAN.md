    1. Create `scripts/scrapers/adapters/pareap/ingest.ts` -- the full PAREAP ingestion orchestrator:
       - Export `async function ingestPareap(): Promise<ScrapeResult>`
       - Creates admin Supabase client via createAdminClient()
       - Ensures PAREAP source exists in `sources` table: upsert { name: "PAREAP", slug: "pareap", base_url: "https://www.pareap.net", scraper_type: "cheerio", is_active: true }. Get back the source UUID.
       - Creates scrape log entry via createScrapeLog(supabase, sourceId)
       - Instantiates PareapAdapter and calls scrape() to get ScrapedJob[]
       - For each scraped job (process in batches of 25 for partial failure resilience):
         a. Call findOrCreateSchool(supabase, job.schoolName, job.city, job.state, job.zipCode, job.schoolType) to get school_id
         b. Upsert into `jobs` table with onConflict: 'source_id,external_id'. Map ScrapedJob fields to jobs table columns. Set is_active=true, last_verified_at=now(). Do NOT set location (geography) -- geocoding deferred to Phase 4.
         c. Upsert into `job_sources` table with onConflict: 'job_id,source_id' for source attribution
         d. Track stats: count added (new inserts), updated (conflict updates), failed (errors)
       - On any batch error: log error to ScrapeError[], continue with next batch (partial failure per user decision DATA-13)
       - Update source.last_scraped_at on the sources table
       - Update scrape log with final stats via updateScrapeLog()
       - Console.log a summary: "PAREAP scrape complete: X added, Y updated, Z skipped, W failed"
       - Return ScrapeResult

    2. Create `scripts/scrapers/run.ts` -- CLI entrypoint:
       - Parse command line args: first positional arg is adapter name (e.g., "pareap")
       - Load dotenv for local development (import 'dotenv/config' or similar)
       - Map adapter name to ingestion function: "pareap" -> ingestPareap
       - Call the ingestion function
       - Log results to console with clear formatting
       - Exit with code 1 if total failures (0 jobs scraped and errors exist), code 0 otherwise
       - Handle uncaught errors with clear logging and exit code 1

       - Add `"scrape:pareap": "npx tsx scripts/scrapers/run.ts pareap"`
       - Ensure tsx is in devDependencies (should be from Plan 01)

       - Ensure `SUPABASE_URL` or `NEXT_PUBLIC_SUPABASE_URL` is listed

    NOTE on distinguishing added vs updated: Supabase upsert does not directly tell you whether a row was inserted or updated.   <action>
    Create `.github/workflows/scrape.yml`:

    ```yaml
    name: Scrape PAREAP Jobs
    on:
      schedule:
        - cron: '0 6 * * *'  # Daily at 6 AM UTC (1-2 AM Eastern)
      workflow_dispatch: {}    # Manual trigger from GitHub UI

    jobs:
      scrape:
        runs-on: ubuntu-latest
        timeout-minutes: 15
        env:
          SUPABASE_URL: ${{ secrets.SUPABASE_URL }}
          SUPABASE_SERVICE_ROLE_KEY: ${{ secrets.SUPABASE_SERVICE_ROLE_KEY }}
          NODE_TLS_REJECT_UNAUTHORIZED: '0'  # PAREAP has SSL cert issues
        steps:
          - uses: actions/checkout@v4
          - uses: actions/setup-node@v4
            with:
              node-version: '20'
              cache: 'npm'
          - run: npm ci
          - run: npm run scrape:pareap
    ```

    Key details:
    - Daily at 6 AM UTC (1-2 AM Eastern) -- off-peak for PAREAP servers
    - workflow_dispatch for manual triggers (testing, on-demand runs)
    - 15-minute timeout (generous for ~30 pages with polite delays)
    - NODE_TLS_REJECT_UNAUTHORIZED=0 for PAREAP's SSL cert issue (scoped to this job only)
    - Secrets referenced: SUPABASE_URL and SUPABASE_SERVICE_ROLE_KEY (user must add these to GitHub repo settings)
    - GitHub Actions sends failure notifications by default when workflow fails (no custom alerting needed per user decision)
